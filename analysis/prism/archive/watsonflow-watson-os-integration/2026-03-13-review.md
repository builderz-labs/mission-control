## New Findings

### Tier 1 (Cross-Validated)

**T1-A: DB file permissions — world-readable (644)**  
Security (cited `stat` output: `644`) + DA (referenced by implication via "any process can bypass auth"):  
`/Users/watson/.openclaw/agents/main/workspace/data/watsonflow.db` is 644 — readable by any process on the machine. Auth in the Next.js API route means nothing if an attacker or rogue agent can open the file directly.  
Fix: `chmod 600 ~/.openclaw/agents/main/workspace/data/watsonflow.db`. Add `fs.chmodSync(DB_PATH, 0o600)` to `db-init.js` and verify launchd plists use `umask 077`.

**T1-B: Auth enforcement not explicitly required on Phase 1 read endpoints**  
Security (cited integration plan wording) + DA (cited Phase 2-only mention of auth): The plan explicitly gates Phase 2 writes with auth, but Phase 1 GETs are ambiguous. Without `requireRole(request, 'viewer')`, the endpoint is open to unauthenticated callers.  
Fix: Add `requireRole` to route handler — same pattern as `src/app/api/tasks/route.ts:73`.

**T1-C: No graceful fallback if WatsonFlow daemon is down**  
Integration (2 blockers cited) + DA (referenced in "rollback/recovery" section): If `watsonflow.db` doesn't exist or is locked, API returns 500. Watson-OS crashes the dashboard panel.  
Fix: `fs.existsSync(dbPath)` check before open; return 503 `{ error: 'WatsonFlow unavailable', available: false }` instead of 500.

### Tier 2 (Single reviewer, specific citation)

**T2-A: WAL IS already enabled** — Performance confirmed via live PRAGMA query:  
`sqlite3 watsonflow.db "PRAGMA journal_mode;"` → `wal`. The R11 PRISM finding (C1) was about a different DB (`mission-control.db`). No action needed.

**T2-B: DB path hardcoded to `/Users/watson/`** — Integration, cited in WATSONFLOW-WATSON-OS-INTEGRATION.md:  
Fix: `WATSONFLOW_DB_PATH` env var in `.env.local`; fallback to `path.join(process.env.HOME, '.openclaw/agents/main/workspace/data/watsonflow.db')`.

**T2-C: Approvals tab duplicates existing `exec-approval-panel.tsx`** — Simplicity, cited `exec-approval-panel.tsx` (400+ lines, full approval UI):  
Fix: Drop Approvals tab from watsonflow-panel. Keep Pipeline + Ideas only. Surface WatsonFlow approvals via filter in existing exec-approval-panel.

**T2-D: 4 API query modes → collapse to 1** — Simplicity:  
`GET /api/watsonflow?view=tasks|ideas|approvals|stats` → single `GET /api/watsonflow` returning `{ tasks, ideas, approvals, counts }`. Client filters. 60% less API code.

**T2-E: Dashboard widget is premature** — Simplicity: Decorative, duplicates panel data. Defer to Phase 2.

**T2-F: Missing read timeout** — Performance: No `busy_timeout` set. If daemon writes large transaction, Watson-OS read blocks indefinitely.  
Fix: `db.pragma('busy_timeout = 3000')` in route handler.

**T2-G: Missing freshness metadata in responses** — Performance: No `last_modified` or `data_age_ms`. Frontend can't detect stale data if daemons crash.  
Fix: Add `meta: { last_modified, data_age_ms, source }` to all responses.

**T2-H: Poll interval unspecified** — Performance: "5-10 seconds" is vague.  
Fix: Default 10s. Support `?poll_ms=` query param. Document rationale in code comments.

**T2-I: Cross-process CAS weakness for Phase 2** — DA: WatsonFlow's CAS guard works within a single process. Watson-OS is a separate process. Simultaneous Discord reaction + web approval creates a race window where both reads see `status='waiting'`, both issue UPDATE.  
Fix: Phase 2 design must use `BEGIN IMMEDIATE` transaction isolation in Watson-OS write path. Document this constraint before Phase 2 code starts.

**T2-J: Schema drift risk** — DA: WatsonFlow schema evolves (11 PRISM rounds), Watson-OS queries will silently miss new/renamed columns.  
Fix: Add `schema_metadata` table to WatsonFlow DB with version key. Watson-OS reads version on startup and logs warning on mismatch.

**T2-K: Panel registration is manual, 3 steps** — Integration, cited `src/app/[[...panel]]/page.tsx` and `navigation.ts:13-18` (no dynamic registry):  
1. Create `watsonflow-panel.tsx`  
2. Import + add to render switch in page.tsx  
3. Add `'watsonflow'` to `DEFAULT_PREFETCH_PANELS` in `navigation.ts`  
Not a blocker — just needs doing.

**T2-L: better-sqlite3 already in package.json** — Integration confirmed: `package.json:14` has `better-sqlite3: ^12.6.2`. No new dependency needed.

## Consensus Points

- Architecture is correct: Watson-OS reads WatsonFlow's SQLite directly in readonly mode. No new servers, no sync jobs, no bidirectional state.
- WAL mode is already enabled — concurrent reads are safe.
- better-sqlite3 is already a dependency — no install needed.
- Phase 1 read-only scope is appropriate. Don't ship writes until Phase 1 is stable.
- Existing `exec-approval-panel.tsx` should handle approvals; new panel handles pipeline + ideas only.
- CAS pattern in WatsonFlow is proven. Phase 2 must extend it, not bypass it.

---

## Contentious Points

**DA said NEEDS WORK; everyone else said AWC.**

DA's core concern: "The plan conflates 'read-only access to a file' with 'safe concurrent access to a database.'" Specifically: WAL lock contention under polling load, and cross-process CAS weakness in Phase 2.

Performance directly refuted the WAL concern: WAL IS enabled, query latencies are 0.2-0.4ms, concurrent readers are safe by design. DA was reviewing as if WAL wasn't enabled — the assumption was wrong.

The cross-process CAS concern (T2-I) is legitimate but Phase 2-scoped, not Phase 1. Phase 1 is read-only — no CAS needed, no write risk.

---

## Conflict Resolution

DA's NEEDS WORK verdict was based on two assumptions: (1) WAL not enabled, (2) Phase 2 CAS risk applies to Phase 1. Performance refuted (1) with live evidence. (2) is real but belongs to Phase 2 design, not a Phase 1 blocker.

**Ruling: Upgrade DA's verdict to AWC.** T2-I (cross-process CAS) becomes a mandatory Phase 2 pre-condition, documented now so it doesn't get missed.

Weight: Performance's Tier 1 live measurement outranks DA's Tier 3 assumption on the WAL question.
